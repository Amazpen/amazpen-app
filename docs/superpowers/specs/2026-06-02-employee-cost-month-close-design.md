# אפיון: סגירת-חודש לעלות עובדים (Employee-Cost Month-Close)

**תאריך:** 2026-06-02
**סטטוס:** מאושר ע"י בעל המוצר (דוד) — מוכן לתוכנית מימוש
**גישה:** A — "פאנל סגירת-חודש" (נבחרה אחרי השוואה ל-B/C)

---

## 1. רקע ובעיה

עלות העובדים בדוח רווח-והפסד ובדשבורד מחושבת היום כ-`(שכר יומי + עלות מנהל) × אחוז העמסה` מתוך `daily_entries`. זו **הערכה** בלבד:
- הסכומים שבאמת יוצאים מהבנק (שכר, פנסיה, ביטוח לאומי, פיצויים) שונים מההערכה ואינם נכנסים לתזרים.
- העלויות הנלוות (פנסיה/סוציאליות/פיצויים) מטופלות "בגדול" דרך אחוז ההעמסה, לא כסכומים אמיתיים ניתנים-למעקב.

**המטרה:** לאפשר לבעל העסק, בסוף החודש, להזין את הסכומים האמיתיים לפי דוח הנהלת-חשבונות, להפוך אותם לחשבוניות בנות-תשלום שנכנסות לתזרים ככל הוצאה, ולגרום לסכום האמיתי לדרוס את ההערכה — בלי כפל-ספירה.

## 2. החלטות מאושרות (מבעל המוצר)

| # | נושא | הכרעה |
|---|------|-------|
| 1 | מקור-אמת | בחודש סגור הסכום האמיתי **דורס** את ההערכה (כולל ההעמסה). הערכה מול בפועל מוצגים זה ליד זה עד הסגירה. |
| 2 | granularity | פיצול לשורות נפרדות: שכר + עלויות מעביד (פנסיה / ביטוח לאומי / פיצויים), כל אחת ספק נפרד. |
| 3 | ספקים מרובים | **חובה:** לרכיב אחד (למשל פנסיה) יכולים להיות כמה ספקים; כולם נכנסים יחד לתשלום בפועל. |
| 4 | העמסה | חלק מההערכה. הזנת הבפועל דורסת אותה. בחודש סגור ההעמסה לא מוחלת. |
| 5 | תצוגה במהלך החודש | הערכת עלויות-המעביד מוצגת כשורה/שורות הערכה נפרדות (פירוק ההעמסה). |
| 6 | היקף | כל עסק שמעסיק עובדים. opt-in-by-action: שום דבר לא משתנה עד שמישהו סוגר חודש. |
| 7 | עיתוי + reopen | סגירה בד"כ עד ה-15 לחודש העוקב; **חובה** אפשרות "פתח מחדש" ולתקן. |
| 8 | צבע | אחרי סגירה+אישור שנכנס לתזרים — המספר מוצג **בירוק**, כמו קונבנציית "שולם/אושר" בשאר האפליקציה. |

## 3. המודל המרכזי — היפוך מקור-אמת

לכל **עסק × חודש** מצב אחד משניים לעלות-עובדים:

- **פתוח (הערכה):** הדוח/דשבורד מציגים `(שכר יומי + מנהל) × העמסה` מ-`daily_entries` — בדיוק כמו היום, בתוספת פירוק-הערכה תצוגתי (סעיף 6).
- **סגור (בפועל):** מקור-האמת עובר לחשבוניות שנוצרו בסגירה. הרכיב היומי (`totalLaborCost`) **מוחרג** לאותו חודש, וההעמסה אינה מוחלת. שורת עלות-העובדים = סכום חשבוניות הסגירה + כל חשבונית `employee_costs` אחרת באותו חודש.

זה מבטל כפל-ספירה: היום הדוח כבר מחבר `totalLaborCost + laborInvoiceActual` (reports/page.tsx:1163-1164). בחודש סגור משמיטים את `totalLaborCost` ומשאירים את חשבוניות הבפועל בלבד.

## 4. סכמת נתונים (migration)

### 4.1 טבלה חדשה `labor_month_close`

| עמודה | טיפוס | הערות |
|-------|-------|-------|
| `id` | uuid PK | `gen_random_uuid()` |
| `business_id` | uuid NOT NULL | FK → businesses |
| `period_year` | int NOT NULL | |
| `period_month` | int NOT NULL | 1-12 |
| `status` | text NOT NULL DEFAULT 'closed' | `'closed'` \| `'reopened'` |
| `estimate_total` | numeric | סנאפשוט ההערכה ברגע הסגירה (לתצוגת הערכה-מול-בפועל) |
| `actual_total` | numeric | סכום הבפועל שהוזן |
| `closed_at` | timestamptz | |
| `closed_by` | uuid | auth user |
| `reopened_at` | timestamptz NULL | |
| `created_at` / `updated_at` | timestamptz DEFAULT now() | |

- `UNIQUE (business_id, period_year, period_month)` — סגירה אחת לחודש.
- **RLS** (4 policies נפרדות SELECT/INSERT/UPDATE/DELETE): `is_business_member(business_id) OR is_admin()`. אין `FOR ALL`.

### 4.2 שינויים בטבלאות קיימות

- `invoices.labor_close_id uuid NULL` — FK → `labor_month_close(id)`. מקשר חשבונית לסגירה שיצרה אותה. מאפשר (א) פתיחה-מחדש מדויקת, (ב) זיהוי חשבוניות-הסגירה ב-P&L. אינדקס על `(labor_close_id)`.
- `suppliers.system_kind text NULL` — לזיהוי ספק-מערכת. ערך `'labor_salary'` לספק "משכורות עובדים" שנוצר אוטומטית פעם אחת לעסק. אינדקס partial-unique: `UNIQUE (business_id, system_kind) WHERE system_kind IS NOT NULL` למניעת כפילות.

> פנסיה / ביטוח לאומי / פיצויים = ספקים רגילים מסוג `employee_costs` שבעל העסק פותח בעצמו בזרימה הקיימת (טאב "עלות עובדים" במסך הספקים). אפשר כמה לכל רכיב (החלטה #3).

## 5. פאנל הסגירה (`LaborMonthCloseModal`)

**נקודת כניסה:** כפתור "סגירת חודש" על שורת "עלות עובדים" בדוח רווח-והפסד (`reports/page.tsx`). אם נבחרו כמה עסקים — דורש בחירת עסק יחיד.

**תוכן:**
- כותרת: `סגירת חודש עלות עובדים — מאי 2026`.
- שורות ממולאות-מראש:
  - **שכר עובדים** ≈ בסיס ההערכה (`rawLabor + computedManager`, בלי העמסה) → ספק-המערכת `system_kind='labor_salary'`.
  - שורה לכל ספק `employee_costs` קיים בעסק (פנסיה/ב"ל/פיצויים), ערך התחלתי 0 או הערכה.
  - **"+ הוסף שורה"** — בחירת ספק `employee_costs` קיים נוסף (תומך כמה קרנות פנסיה — החלטה #3). אפשר גם לפתוח ספק חדש בזרימה הקיימת.
- כל שורה: בורר ספק, סכום-בפועל, מע"מ לפי `effectiveVatType(supplier)` (שכר/פנסיה בד"כ ללא מע"מ — ראה gotcha `requires_vat` vs `vat_type`), תאריך-יעד לתשלום לפי תנאי הספק.
- תחתית: **סה"כ הערכה · סה"כ בפועל · הפרש** (תומך ב"הערכה מול בפועל").
- כפתור **"סגור חודש"**.

**שמירה (POST):** לכל שורה עם סכום>0 נוצרת חשבונית:
- `supplier_id`, `subtotal`/`vat_amount`/`total_amount` לפי `effectiveVatType`,
- `invoice_date` = היום האחרון בחודש הנסגר, `due_date` לפי תנאי הספק,
- `status='pending'`, `labor_close_id`,
- `invoice_number` אוטומטי (למשל `SAL-2026-05`, `PEN-2026-05`),
- `notes` = `"סגירת חודש עלות עובדים 05/2026"`.

נוצרת רשומת `labor_month_close` (status='closed', estimate_total/actual_total, closed_at/by). **V1 מייצר חשבוניות `pending` בלבד** (החלטה: לא "שלם עכשיו"). התשלום מתבצע בזרימה הקיימת (`/expenses` / `/payments`) ומשם נכנס לתזרים.

## 6. פירוק-הערכה במהלך החודש (תצוגתי בלבד)

בחודש **פתוח**, שורת עלות-העובדים מוצגת מפוצלת ל-2 תת-שורות (הסכום הכולל זהה — שינוי ויזואלי בלבד, אפס סיכון מספרי):
- **שכר** ≈ `rawLabor + computedManager`.
- **עלויות מעביד** ≈ דלתת ההעמסה: `(rawLabor + manager) × (avgMarkup − 1)`.

אין פיצול נוסף לפנסיה/ב"ל/פיצויים בשלב ההערכה (אין יחסים מוגדרים; הפיצול האמיתי מגיע בסגירה מדוח הנה"ח).

## 7. אינטגרציה ב-P&L ובדשבורד (השינוי הקריטי)

ב-`reports/page.tsx` (סביב שורה 1164) וב-`page.tsx` (דשבורד, חישוב זהה):

1. קביעת מצב החודש המוצג: query ל-`labor_month_close` עבור business+year+month, `status='closed'`.
2. **חודש סגור:**
   - `parentActual` של עלות-עובדים = `laborInvoiceActual` בלבד (סכום חשבוניות `employee_costs` באותו חודש) — **בלי** `totalLaborCost`.
   - ב-`allExpensesActual` (reports:1199) — לאפס את תרומת `totalLaborCost` לאותו חודש.
   - הסכום מוצג **בירוק** (קונבנציית "שולם/אושר", החלטה #8).
3. **חודש פתוח:** ללא שינוי מספרי + פירוק-הערכה תצוגתי (סעיף 6).

**אימות-מימוש חובה:** לוודא שחשבוניות `employee_costs` אינן נספרות גם ב-`totalCurrentExpenses` (לפי ההערה reports:1196-1198 רק `current_expenses` נספרים שם — לאמת בפועל לפני סיום).

## 8. פתיחה-מחדש (reopen)

כפתור "פתח מחדש" על חודש סגור → מודאל-אישור בדפוס destructive (מציג בדיוק מה יבוטל — ראה pattern `destructive_save_with_confirm`).

- אם לאחת מחשבוניות הסגירה (`labor_close_id`) יש תשלום מקושר (`payment_invoice_links`) → **חוסם** עם הסבר ("בטל קודם את התשלום המקושר"), כדי לא לפגוע בתזרים.
- אחרת: מוחק את חשבוניות הסגירה, מסמן `labor_month_close.status='reopened'`, `reopened_at=now()`. ה-P&L חוזר אוטומטית להערכה היומית. ניתן לסגור שוב (re-close מעדכן את הרשומה חזרה ל-'closed').

## 9. שרת — API route ייעודי

`/api/labor-close`:
- **POST** — סגירת חודש: יצירת חשבוניות מרובות + רשומת הסגירה, **אטומי** (טרנזקציה), RLS-safe. קלט: `business_id`, `year`, `month`, `lines[] = {supplier_id, amount, vat_type?, due_date?}`, `estimate_total`.
- **DELETE** — פתיחה-מחדש: בדיקת תשלומים מקושרים → חסימה/מחיקה + עדכון סטטוס.

הכתיבה בשרת (לא multi-insert מהקליינט) לאטומיות ולמניעת מצב-ביניים שבור.

## 10. יחידות עבודה (לפירוק לתוכנית מימוש)

1. **DB migration** — טבלה `labor_month_close` + RLS (4 policies) + `invoices.labor_close_id` + `suppliers.system_kind` + אינדקסים.
2. **ספק-מערכת "משכורות"** — provision-on-first-close (`system_kind='labor_salary'`, `expense_type='employee_costs'`, ללא מע"מ).
3. **API route** `/api/labor-close` (POST/DELETE) — לוגיקת יצירה/ביטול אטומית.
4. **`LaborMonthCloseModal`** — פאנל הסגירה (pre-fill, שורות מרובות, הערכה-מול-בפועל).
5. **P&L flip** ב-`reports/page.tsx` — קריאת מצב סגירה + היפוך מקור-אמת + צבע ירוק + פירוק-הערכה.
6. **Dashboard flip** ב-`page.tsx` — אותו היפוך.
7. **כפתור reopen** + מודאל-אישור.

## 11. סיכונים ואי-רגרסיה

- **עסקים קיימים:** ללא סגירה — אפס שינוי (opt-in-by-action). אין gate לפי `business_type`.
- **כפל-ספירה:** נמנע ע"י היפוך מקור-האמת + אימות `totalCurrentExpenses` (סעיף 7).
- **תזרים:** חשבוניות הסגירה זורמות דרך `payments`/`payment_splits` הקיימים; אין צינור תזרים חדש.
- **RLS:** כל טבלת-כתיבה חדשה עם `is_business_member() OR is_admin()` בפיצול 4 פעולות.

## 12. מחוץ להיקף (V1)

- "שלם עכשיו" בתוך הפאנל (V1 מייצר payables בלבד).
- פיצול אוטומטי של ההעמסה לפנסיה/ב"ל/פיצויים לפי יחסים (אין יחסים מוגדרים).
- עריכת סכומי חודש-סגור בלי reopen מלא.
- היסטוריית-גרסאות לסגירות (מעבר ל-closed/reopened).
